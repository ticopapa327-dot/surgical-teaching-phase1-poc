# Phase 2 呼叫与互动 PoC 交付交接说明

## 一、当前结论

Phase 2 已形成可运行、可自动化验证、可持续迭代的呼叫与互动控制面 PoC。当前版本适合继续进入真实媒体链路 PoC、跨终端联调和业务集成设计；但不能直接作为生产级手术示教系统交付使用。

## 二、已完成能力

1. 手术室端、示教室端和观摩端可通过 WebSocket 信令服务完成终端注册、在线目录、呼叫、接受、拒绝、取消、会话建立、会话结束和离会。
2. 支持交互模式和仅收看模式；任一方选择仅收看时，最终会话按仅收看处理。
3. 支持会议参与上限，且上限和会话 owner 优先按手术室端归属解析。
4. 支持会话目录发现，观摩端可从目录选择或手工输入会话 ID 加入会议。
5. 支持按角色上报能力和通道，观摩端不会声明手术室端 4 路发布通道。
6. 支持会话内订阅通道更新、标注同步和 `peer.signal` 协商消息透传。
7. 支持忙线保护、待处理呼叫超时、重复 endpointId 接管、会话恢复、协议错误分支和 WebSocket 心跳清理。
8. 支持可选共享令牌门禁、HTTP 健康检查、会话摘要目录和内存事件日志。
9. 前端支持信令断线时清理会话、在线目录、目标选择、健康状态和本地交互音频轨道。
10. 远端画面可打开独立扩展窗口；Electron 模式可枚举 Windows 显示器并将扩展窗口定位到指定显示器。

## 三、验证状态

统一验证命令：

```powershell
npm run verify
```

该命令覆盖：

1. Vite 生产构建。
2. 高危依赖审计。
3. Playwright 前端烟测。
4. WebSocket 信令契约测试。

当前 GitHub Actions `CI / verify` 已接入上述验证流程，main 分支推送后自动执行。

## 四、仍未实现边界

1. 真实跨终端音视频媒体传输、WebRTC/SFU、WHEP、RTSP/SRT 接入和媒体转码。
2. Android 会议平板原生客户端。
3. 手机端直播分发和多并发直播服务器。
4. 非 UVC 云台控制、厂商私有协议和厂商 SDK 适配。
5. 真实 HIS 联网、接口鉴权、字段映射和患者信息权限控制。
6. FTP 上传、移动存储导出流程、录像归档策略和文件生命周期管理。
7. 生产级终端鉴权、TLS、审计、权限模型和操作日志。
8. 生产级多屏布局配置、窗口生命周期管理和启动恢复。

## 五、建议下一阶段

下一阶段应优先验证真实媒体链路，不建议继续堆叠前端模拟效果。建议顺序如下：

1. 建立媒体服务 PoC，明确 WebRTC/SFU 与 RTSP/SRT 接入边界。
2. 验证手术室端 4 路媒体发布、示教室端按需订阅和观摩端加入后的资源占用。
3. 在真实 USB 采集卡和医院局域网环境中复测延迟、丢包恢复、音频回声消除和长时间稳定性。
4. 将信令会话 ID 与媒体房间 ID、通道 ID 建立明确映射。
5. 媒体链路稳定后，再进入 HIS、FTP、AI 任务、Android 会议平板和手机直播分发。
