# Phase 3 按订阅 WebRTC 媒体 PoC 说明

## 一、目标

本阶段目标是补齐“信令已建立但示教室端黑屏”和“音频未进入远端链路”的最小媒体闭环，并从单通道推进到按订阅多通道：在已建立信令会话后，由手术室端按远端订阅发布通道 1 至通道 4 视频，交互模式下双方尽量将本地麦克风加入 WebRTC，并通过浏览器 P2P 接收远端 MediaStream。

## 二、已实现能力

1. 复用现有 WebSocket 信令服务的 `peer.signal` 消息透传。
2. 手术室端点击“发布订阅通道媒体”后，为会话内远端参与方创建 `RTCPeerConnection`。
3. 手术室端根据每个远端在会话中的 `subscriptions` 加入对应通道的视频轨道；交互模式下同时尝试加入本地麦克风轨道。
4. 发布中远端订阅集合变化时，手术室端会自动按新的订阅集合重新协商媒体链路。
5. 示教室端收到 offer 后创建 answer；交互模式下也尝试加入本地麦克风轨道，并通过同一信令会话回传。
6. 双方通过 `peer.signal` 交换 ICE candidate。
7. 示教室端收到远端 video track 后，根据 `peer.signal` 中的 `tracks` 映射将远端显示区对应通道的视频元素切换为真实远端 MediaStream，避免多通道串路。
8. 双方收到远端 audio track 后，使用独立 audio sink 播放远端音频，避免视频元素 muted 导致无声。
9. 交互音频启用低延迟采集约束、Opus 低延迟 SDP 参数、`ptime/maxptime`、音频 RTP 高优先级和接收端 playout/jitter buffer 优化。
10. 前端状态区显示 WebRTC 统计，包括音频缓冲、jitter 和 RTT，用于判断延迟来源。
11. 媒体诊断区显示每路远端媒体状态，以及每个远端 PeerConnection 的连接、ICE 和协商状态。
12. 断开信令、离会、结束会话或点击“停止媒体链路”时清理 PeerConnection 和远端流状态。

## 三、当前边界

1. 当前验证浏览器 P2P 按订阅发布多路模拟视频和基础交互音频；真实 USB 4 路采集卡并发、长稳运行和资源占用仍需专项联调。
2. 只验证两端浏览器 P2P，不包含 SFU、多方媒体转发或手机直播分发。
3. 默认 `webrtc.iceServers` 为空，适合医院同一局域网测试；跨网段、NAT 或 VPN 场景需要在 `public/config.json` 配置 STUN/TURN 后重新验证。
4. 本地麦克风采集受浏览器安全上下文限制；从局域网 HTTP 页面打开的示教室端可能只能接收远端音频，不能采集本地麦克风发言。双端发言建议使用本机 `127.0.0.1`、Electron 客户端或 HTTPS。
5. 蓝牙耳机、电视扬声器、HDMI 音频处理、扩展坞声卡和系统级音效增强可能带来明显播放延迟；现场低延迟测试应优先使用有线麦克风和有线扬声器。
6. RTSP、SRT、USB 采集卡驱动兼容性和云台控制仍是后续专项。

## 四、手动验证步骤

1. 两台电脑按 `docs/two-pc-test-manual.md` 启动信令服务和前端。
2. 手术室端和示教室端分别连接同一个信令地址。
3. 任一端发起呼叫，另一端接受呼叫，确认双方进入会话。
4. 示教室端按需勾选通道 1 至通道 4，并选择单画面、双画面或四画面布局。
5. 手术室端点击“发布订阅通道媒体”；未启动预览的订阅通道会自动启动模拟源预览。
6. 示教室端“状态”区的“媒体链路”应显示正在接收或已收到手术室端媒体。
7. 示教室端远端显示区应出现已订阅通道的动态画面。
8. 交互模式下，双方“交互音频”状态应显示低延迟本地音频已加入或已接收远端音频。
9. “媒体统计”中音频缓冲长期高于 200 ms 或 RTT 明显异常时，应先排查浏览器、音频设备和网络。

## 五、自动化验证

执行：

```powershell
npm run verify
```

其中 Playwright 烟测包含双页面和三页面 WebRTC 用例：启动真实信令服务，模拟手术室端、示教室端和观摩端完成呼叫、差异化订阅、动态追加订阅、发布后观摩端加入、协商、多个远端视频轨道接收和双端远端音频轨道接收。

## 六、下一步建议

1. 在真实 USB 采集卡输入上复测多通道 WebRTC 延迟、码率和连续运行稳定性。
2. 记录 1/2/3/4 路发布时的 CPU、内存、显存、带宽占用和丢帧情况。
3. 在真实双机环境复测音频端到端延迟、回声消除、降噪、自动增益、扬声器回放和非安全 HTTP 页面限制。
4. 引入 SFU PoC，验证多人会议和手机直播由独立媒体服务器承担。
5. 再接入 RTSP/SRT 输入适配和媒体服务侧转码。
