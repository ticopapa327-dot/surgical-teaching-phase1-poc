# Phase 3 单路 WebRTC 媒体 PoC 说明

## 一、目标

本阶段目标是补齐“信令已建立但示教室端黑屏”和“音频未进入远端链路”的最小媒体闭环：在已建立信令会话后，由手术室端发布通道 1 视频，交互模式下双方尽量将本地麦克风加入 WebRTC，并通过浏览器 P2P 接收远端 MediaStream。

## 二、已实现能力

1. 复用现有 WebSocket 信令服务的 `peer.signal` 消息透传。
2. 手术室端点击“发布通道 1 媒体”后，为会话内远端参与方创建 `RTCPeerConnection`。
3. 手术室端将本地通道 1 预览流的视频轨道加入 WebRTC 连接；交互模式下同时尝试加入本地麦克风轨道。
4. 示教室端收到 offer 后创建 answer；交互模式下也尝试加入本地麦克风轨道，并通过同一信令会话回传。
5. 双方通过 `peer.signal` 交换 ICE candidate。
6. 示教室端收到远端 video track 后，将远端显示区对应通道的视频元素切换为真实远端 MediaStream。
7. 双方收到远端 audio track 后，使用独立 audio sink 播放远端音频，避免视频元素 muted 导致无声。
8. 断开信令、离会、结束会话或点击“停止媒体链路”时清理 PeerConnection 和远端流状态。

## 三、当前边界

1. 只验证通道 1 视频和基础交互音频，不验证 4 路并发远端发布。
2. 只验证两端浏览器 P2P，不包含 SFU、多方媒体转发或手机直播分发。
3. 当前未配置 STUN/TURN，适合医院同一局域网测试；跨网段、NAT 或 VPN 场景不保证可用。
4. 本地麦克风采集受浏览器安全上下文限制；从局域网 HTTP 页面打开的示教室端可能只能接收远端音频，不能采集本地麦克风发言。双端发言建议使用本机 `127.0.0.1`、Electron 客户端或 HTTPS。
5. RTSP、SRT、USB 采集卡驱动兼容性和云台控制仍是后续专项。

## 四、手动验证步骤

1. 两台电脑按 `docs/two-pc-test-manual.md` 启动信令服务和前端。
2. 手术室端和示教室端分别连接同一个信令地址。
3. 任一端发起呼叫，另一端接受呼叫，确认双方进入会话。
4. 手术室端先启动通道 1 预览；如果未启动，点击“发布通道 1 媒体”时会自动启动。
5. 手术室端点击“发布通道 1 媒体”。
6. 示教室端“状态”区的“媒体链路”应显示正在接收或已收到手术室端通道 1。
7. 示教室端远端显示区应出现手术室端通道 1 的动态画面。
8. 交互模式下，双方“交互音频”状态应显示本地音频已加入或已接收远端音频。

## 五、自动化验证

执行：

```powershell
npm run verify
```

其中 Playwright 烟测包含双页面 WebRTC 用例：启动真实信令服务，模拟手术室端和示教室端完成呼叫、协商、远端视频轨道接收和双端远端音频轨道接收。

## 六、下一步建议

1. 在真实 USB 采集卡输入上复测通道 1 WebRTC 延迟、码率和连续运行稳定性。
2. 扩展为 4 路视频发布与按需订阅，并记录 CPU、内存、带宽占用。
3. 在真实双机环境复测音频回声消除、降噪、自动增益、扬声器回放和非安全 HTTP 页面限制。
4. 引入 SFU PoC，验证多人会议和手机直播由独立媒体服务器承担。
5. 再接入 RTSP/SRT 输入适配和媒体服务侧转码。
