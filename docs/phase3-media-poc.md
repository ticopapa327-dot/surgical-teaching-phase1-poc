# Phase 3 按订阅 WebRTC 媒体 PoC 说明

## 一、目标

本阶段目标是补齐“信令已建立但示教室端黑屏”和“音频未进入远端链路”的最小媒体闭环，并从单通道推进到按订阅多通道：在已建立信令会话后，由手术室端按远端订阅发布通道 1 至通道 4 视频，交互模式下双方尽量将本地麦克风加入 WebRTC，并通过浏览器 P2P 接收远端 MediaStream。

## 二、已实现能力

1. 复用现有 WebSocket 信令服务的 `peer.signal` 消息透传，并在透传消息与事件摘要中携带 `mediaRoomId`。
2. 手术室端点击“发布订阅通道媒体”后，为会话内远端参与方创建 `RTCPeerConnection`。
3. 手术室端根据每个远端在会话中的 `subscriptions` 加入对应通道的视频轨道；交互模式下同时尝试加入本地麦克风轨道。
4. 发布中远端订阅集合、参与方或远端 endpoint 注册版本发生变化时，手术室端会自动按新的会话状态重新协商媒体链路。
5. 示教室端收到 offer 后创建 answer；交互模式下也尝试加入本地麦克风轨道，并通过同一信令会话回传。
6. 双方通过 `peer.signal` 交换 ICE candidate。
7. 示教室端收到远端 video track 后，根据 `peer.signal` 中的 `tracks` 映射将远端显示区对应通道的视频元素切换为真实远端 MediaStream，避免多通道串路。
8. 双方收到远端 audio track 后，使用独立 audio sink 播放远端音频，避免视频元素 muted 导致无声。
9. 音频面板支持选择麦克风输入和远端音频回放输出设备；输出选择依赖浏览器 `setSinkId` 能力。
10. 交互音频启用低延迟采集约束、Opus 低延迟 SDP 参数、`ptime/maxptime`、音频 RTP 高优先级和接收端 playout/jitter buffer 优化。
11. 前端状态区显示 WebRTC 统计，包括视频发送/接收码率、视频包发/收/丢、音频缓冲、jitter、RTT 和 ICE 候选类型路由，用于判断延迟来源和是否走 TURN/relay。
12. 前端状态区显示音频诊断，包括安全上下文、采集能力、输入/输出设备数量、音频输出选择能力和当前麦克风/回放选择。
13. 媒体诊断区显示每路远端媒体状态，以及每个远端 PeerConnection 的连接、ICE、协商状态、本地发送音频轨道数和远端接收音频轨道数。
14. 示教室端或观摩端可点击“请求重发媒体”，通过 `media-refresh-request` 请求手术室端按当前订阅重新发布媒体。
15. 断开信令、离会、结束会话或点击“停止媒体链路”时清理 PeerConnection 和远端流状态；手术室 owner 停止媒体时通知所有远端，示教室端或观摩端停止媒体时只通知手术室 owner 清理本端对应链路，不影响其他接收端。
16. 状态区提供“复制诊断快照”，生成浏览器运行环境、本端控制面、音频、媒体、通道源模式、预览状态、结构化 WebRTC 统计、PeerConnection 连接/ICE/协商状态和最近信令事件摘要；已连接信令时会先限量刷新最近 10 条事件日志。快照包含会话、会话事件摘要和 `peer.signal` 摘要中的 `mediaRoomId`，不包含信令令牌、设备 ID、SDP、ICE candidate 原文、患者信息或媒体数据。

## 三、当前边界

1. 当前验证浏览器 P2P 按订阅发布多路模拟视频和基础交互音频；真实 USB 4 路采集卡并发、长稳运行和资源占用仍需专项联调。
2. 只验证少量浏览器 P2P 链路，不包含 SFU、生产级多方媒体转发或手机直播分发。
3. 默认 `webrtc.iceServers` 为空，适合医院同一局域网测试；跨网段、NAT 或 VPN 场景需要参考 `public/config.example.json` 在 `public/config.json` 配置 STUN/TURN 后重新验证，且不得提交真实 TURN 凭据。
4. 本地麦克风采集受浏览器安全上下文限制；从局域网 HTTP 页面打开的示教室端可能只能接收远端音频，不能采集本地麦克风发言。双端发言建议使用本机 `127.0.0.1`、Electron 客户端或 HTTPS。
5. 音频输出设备选择依赖 Chromium 系浏览器的 `setSinkId`；不支持该能力时，远端音频继续走系统默认播放设备。
6. 蓝牙耳机、电视扬声器、HDMI 音频处理、扩展坞声卡和系统级音效增强可能带来明显播放延迟；现场低延迟测试应优先使用有线麦克风和有线扬声器。
7. RTSP、SRT、USB 采集卡驱动兼容性和非 UVC/厂商 SDK 云台控制仍是后续专项。

## 四、手动验证步骤

1. 两台电脑按 `docs/two-pc-test-manual.md` 启动信令服务和前端。
2. 手术室端和示教室端分别连接同一个信令地址。
3. 任一端发起呼叫，另一端接受呼叫，确认双方进入会话。
4. 示教室端按需勾选通道 1 至通道 4，并选择单画面、双画面或四画面布局。
5. 手术室端点击“发布订阅通道媒体”；未启动预览的订阅通道会自动启动模拟源预览。
6. 示教室端“状态”区的“媒体链路”应显示正在接收或已收到手术室端媒体。
7. 示教室端远端显示区应出现已订阅通道的动态画面。
8. 交互模式下，双方“交互音频”状态应显示低延迟本地音频已加入或已接收远端音频；“音频诊断”应显示安全上下文、输入/输出设备数量和当前回放设备。
9. 如果示教室端黑屏或媒体诊断长期停留在等待状态，可点击“请求重发媒体”，手术室端应自动重新协商并发布当前订阅通道。
10. 示教室端或观摩端点击“停止媒体链路”后，本端远端画面应清空，手术室端只关闭该终端对应的 PeerConnection；其他仍在线接收端应继续显示已订阅画面。
11. “媒体统计”中视频发送或接收码率长期为 `-`、音频缓冲长期高于 200 ms、RTT 明显异常或 ICE 路由不是预期的 `host`/`relay` 时，应先排查浏览器、音频设备、网络和 STUN/TURN 配置。
12. 出现黑屏、音频无声或延迟异常时，两端分别点击“复制诊断快照”，保存快照文本作为排障依据，重点比对 `signaling`、`session.mediaRoomId`、`channels`、`media.stats`、`media.statsMetrics`、`media.peerConnections` 和 `recentEvents` 中的 `mediaRoomId`。

## 五、自动化验证

执行：

```powershell
npm run verify
```

其中 Playwright 烟测包含双页面和三页面 WebRTC 用例：启动真实信令服务，模拟手术室端、示教室端和观摩端完成呼叫、差异化订阅、动态追加订阅、发布后观摩端加入、观摩端停止自身媒体链路且不影响其他接收端、观摩端离会后重发布、观摩端异常断开后重发布、示教室端同 ID 重连后重发布、远端请求媒体重发、协商、多个远端视频轨道接收和双端远端音频轨道接收。

现场保存两端诊断快照后，可执行 `node scripts/summarize-diagnostics.cjs snapshot-a.json snapshot-b.json > diagnostics.csv` 汇总会话、媒体房间、订阅通道数、已到达/等待/异常通道数、PeerConnection、码率、音频缓冲、RTT 和 ICE 路由；也可执行 `npm run diagnostics:analyze -- snapshot-a.json snapshot-b.json` 自动提示媒体房间不一致、非安全上下文、缺少 `peer.signal`、订阅通道诊断未到齐、远端画面等待、PeerConnection 异常、无视频码率、音频缓冲过高或 RTT 过高等常见问题。手术室端处于发布状态时，分析器不会把本端默认远端画面等待误判为接收端黑屏。
需要把快照分析纳入自动化复测时，可增加 `--fail-on-warn`，出现 `WARN` 时脚本返回非零退出码。
如果复测是仅收看模式，并且接收端通过 PC-A 的局域网 HTTP 地址打开页面，可额外使用 `--allow-receive-only-runtime-warn`，将接收端非安全上下文和本地麦克风不可用降为 `INFO`；该开关不放宽 WebRTC、视频码率、订阅通道、PeerConnection 或媒体房间一致性检查。
117 Windows 远程浏览器已支持本地自动化媒体 smoke：PC-A 执行 `npm run test:remote:media` 可通过 117 的 Edge DevTools 完成 4 路订阅、发布媒体、远端 live 检查、双端快照保存和诊断分析。媒体 smoke 通过后，执行 `npm run test:remote:diagnostics` 会自动读取最近一组 118/117 快照，生成 `*-diagnostics.csv` 并再次运行带门禁的诊断分析；也可通过 `npm run test:remote:diagnostics -- snapshot-a.json snapshot-b.json` 分析指定快照。

## 六、下一步建议

1. 在真实 USB 采集卡输入上复测多通道 WebRTC 延迟、码率和连续运行稳定性。
2. 记录 1/2/3/4 路发布时的 CPU、内存、显存、带宽占用和丢帧情况。
3. 在真实双机环境复测音频端到端延迟、回声消除、降噪、自动增益、扬声器回放和非安全 HTTP 页面限制。
4. 引入 SFU PoC，验证多人会议和手机直播由独立媒体服务器承担。
5. 再接入 RTSP/SRT 输入适配和媒体服务侧转码。
