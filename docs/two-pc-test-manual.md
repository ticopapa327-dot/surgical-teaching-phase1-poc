# 两台电脑手术室与示教室模拟测试使用手册

## 一、适用范围

本文档用于在两台电脑上验证当前阶段手术示教软件 PoC 的呼叫、互动、会话目录、按需拉流控制、标注、会议上限、断线清理、基础录像流程和按订阅 WebRTC 音视频链路。

当前阶段已支持手术室端按示教室端订阅发布通道 1 至通道 4 的浏览器 WebRTC P2P 视频，并在交互模式下尝试建立 WebRTC 音频。该能力仍是 PoC：暂不包含 SFU、多方媒体转发、RTSP/SRT、Android 会议平板和手机直播。

## 二、本阶段开发成果汇总

1. 完成 4 路本地预览、选择性录制、录像索引、基础回放、搜索、导出、FTP 上传和删除。
2. 完成 WebSocket 信令服务，支持终端注册、在线目录、呼叫、接受、拒绝、取消、会话建立、会话结束、离会和会话目录发现。
3. 支持手术室端与示教室端双向发起呼叫，支持交互模式和仅收看模式确认。
4. 支持会议参与上限，超限时服务端拒绝加入并在前端提示。
5. 支持手术室端会话 owner 归属，多人会议中非 owner 不能结束整场会议。
6. 支持按需订阅通道、布局切换、标注广播和 `peer.signal` 协商消息透传。
7. 支持可选共享令牌门禁、HTTP 健康检查、会话摘要目录和内存事件日志。
8. 支持信令忙线保护、待处理呼叫超时、重复 endpointId 接管、会话恢复、远端同 ID 重连后的媒体重协商、协议错误分支和 WebSocket 心跳清理。
9. 支持信令断开时清理会话、在线目录、目标选择、健康状态和本地交互音频轨道。
10. 支持远端画面独立扩展窗口；Electron 模式下可枚举 Windows 显示器并定位到指定显示器。
11. 支持手术室端在信令会话内按远端订阅发布 WebRTC 视频，示教室端远端显示区接收真实远端 MediaStream。
12. 支持发布中订阅变化、观摩端加入、观摩端离会、普通观摩端异常断开或示教室端同 ID 重连后由手术室端自动重新协商媒体链路。
13. 支持交互模式下将本地麦克风加入 WebRTC 链路，并使用独立 audio sink 播放远端音频。
14. 已接入 GitHub Actions CI，`npm run verify` 覆盖构建、高危依赖审计、前端烟测、双页面和三页面 WebRTC 音视频用例以及信令契约测试。

## 三、测试拓扑

推荐拓扑如下：

| 角色 | 电脑 | 运行内容 | 访问地址 |
|---|---|---|---|
| 手术室端 | PC-A | 信令服务器、前端客户端 | `http://127.0.0.1:5173` |
| 示教室端 | PC-B | 前端客户端 | `http://127.0.0.1:5173` |
| 信令服务 | PC-A | WebSocket 服务 | `ws://<PC-A局域网IP>:7077/signal` |

推荐两台电脑都本地运行前端。原因是浏览器对摄像头、麦克风等媒体权限通常要求安全上下文，`127.0.0.1` 比直接访问另一台电脑的 `http://<IP>:5173` 更稳定。

## 四、测试前准备

### 1. 软件环境

推荐两台电脑均具备：

1. Windows 10、Windows 11 或可运行现代 Chrome/Edge 的 macOS。
2. Node.js 与 npm。
3. Chrome 或 Edge。
4. 本仓库代码和依赖。

在两台电脑的仓库目录执行：

```powershell
cd D:\CodeX\UST\phase1-poc
npm install
```

如果已经安装过依赖，可以跳过 `npm install`。

### 2. 获取手术室端电脑 IP

在 PC-A 执行：

```powershell
ipconfig
```

记录当前局域网网卡的 IPv4 地址，例如：

```text
192.168.10.21
```

后文用 `<PC-A局域网IP>` 表示该地址。

### 3. 开放信令端口

在 PC-A 以管理员身份打开 PowerShell，执行：

```powershell
New-NetFirewallRule -DisplayName "UST Signaling 7077" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 7077
```

如果需要从 PC-B 直接访问 PC-A 的前端页面，还需要开放 `5173`：

```powershell
New-NetFirewallRule -DisplayName "UST Web 5173" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 5173
```

推荐测试方式不要求开放 `5173`，因为两台电脑各自运行本地前端。

## 五、启动步骤

### 1. 在 PC-A 启动信令服务器

推荐在 PC-A 使用一键局域网启动：

```powershell
cd D:\CodeX\UST\phase1-poc
npm run dev:lan:check
npm run dev:lan
```

`dev:lan:check` 只检查端口占用并打印可访问地址，不启动服务。确认端口检查通过后再执行 `npm run dev:lan`，该命令会同时启动信令服务和前端页面。正常输出会包含：

```text
UST LAN test service configuration
Signaling bind: ws://0.0.0.0:7077/signal
Web bind:       http://0.0.0.0:5173
Adapter: 以太网 2 (recommended)
  LAN URL: http://192.168.1.118:5173
  Signal:  ws://192.168.1.118:7077/signal
Port check: OK
Services ready. Keep this terminal open during the two-PC test.
```

如果 PC-A 有多块网卡，可用网卡名称片段优先推荐有线网卡，例如：

```powershell
$env:UST_PREFERRED_ADAPTER="以太网"
npm run dev:lan:check
npm run dev:lan
```

该设置只影响输出中推荐地址的排序；服务仍默认绑定 `0.0.0.0`，不会限制其他网卡访问。

开发自检时可使用 `npm run dev:lan:smoke`，该命令启动服务并在 ready 探测通过后自动退出；正式双机测试仍使用 `npm run dev:lan` 并保持窗口打开。

如果需要分开启动，也可以在 PC-A 执行：

```powershell
cd D:\CodeX\UST\phase1-poc
$env:SIGNALING_HOST = "0.0.0.0"
$env:SIGNALING_PORT = "7077"
npm run server:signaling
```

正常输出应包含：

```text
signaling server listening on ws://0.0.0.0:7077/signal
```

如果需要启用共享令牌门禁：

```powershell
$env:SIGNALING_AUTH_TOKEN = "test-token"
```

启用令牌后，两台前端都必须填写同一个信令令牌。

如需跨网段、VPN 或 TURN 中继联调，可从示例文件复制运行时配置后再修改：

```powershell
Copy-Item public\config.example.json public\config.json
```

`public/config.example.json` 只包含占位 STUN/TURN 地址和凭据。真实 TURN 用户名、密码、医院内网密钥或公网中继地址不应提交到公开仓库。

如需在 PC-A 手术室端验证 FTP 上传，应使用 Electron 客户端并在启动前设置 FTP 环境变量。浏览器模式、`npm run dev:web` 和 `npm run dev:lan` 只能验证按钮提示，不能直接上传本地录像文件。

```powershell
$env:UST_FTP_HOST = "192.168.1.10"
$env:UST_FTP_PORT = "21"
$env:UST_FTP_USER = "ftp-user"
$env:UST_FTP_PASSWORD = "ftp-password"
$env:UST_FTP_REMOTE_DIR = "/ust-recordings"
npm run dev
```

### 2. 在 PC-A 启动手术室端前端

如果已使用 `npm run dev:lan`，本步骤跳过。分开启动时，在 PC-A 新开一个 PowerShell 窗口：

```powershell
cd D:\CodeX\UST\phase1-poc
npm run dev:web
```

浏览器打开：

```text
http://127.0.0.1:5173
```

### 3. 在 PC-B 启动示教室端前端

在 PC-B 执行：

```powershell
cd D:\CodeX\UST\phase1-poc
npm run dev:web
```

浏览器打开：

```text
http://127.0.0.1:5173
```

### 4. 可选：只在 PC-A 托管前端页面

如果 PC-B 不安装仓库，可以在 PC-A 用局域网方式启动前端：

```powershell
cd D:\CodeX\UST\phase1-poc
npx vite --host 0.0.0.0 --port 5173
```

PC-B 浏览器打开：

```text
http://<PC-A局域网IP>:5173
```

该方式适合示教室端收看和验证信令控制流。由于浏览器可能阻止非安全 HTTP 来源的摄像头和麦克风权限，不推荐把手术室端采集发布也放在远程 HTTP 页面上测试。

说明：当前版本会根据页面地址自动修正默认信令地址。PC-A 使用 `http://127.0.0.1:5173` 时默认连接 `ws://127.0.0.1:7077/signal`；PC-B 使用 `http://192.168.1.118:5173` 时默认连接 `ws://192.168.1.118:7077/signal`。默认本端 ID 也会在浏览器本地生成唯一值，避免两端都使用 `or-local` 导致在线目录只剩一个终端。

## 六、客户端配置

### 1. PC-A 手术室端配置

在前端“信令控制面”中填写：

| 字段 | 建议值 |
|---|---|
| 信令地址 | `ws://<PC-A局域网IP>:7077/signal` |
| 信令令牌 | 为空；如启用门禁则填写 `test-token` |
| 本端 ID | `or-room-1` |
| 本端名称 | `手术室端 PC-A` |
| 本端角色 | `手术室端` |

点击“连接信令”。预期显示“已注册 手术室端 PC-A”。

### 2. PC-B 示教室端配置

在前端“信令控制面”中填写：

| 字段 | 建议值 |
|---|---|
| 信令地址 | `ws://<PC-A局域网IP>:7077/signal` |
| 信令令牌 | 为空；如启用门禁则填写 `test-token` |
| 本端 ID | `teach-room-1` |
| 本端名称 | `示教室端 PC-B` |
| 本端角色 | `示教室端` |

点击“连接信令”。预期显示“已注册 示教室端 PC-B”。

### 3. 连通性确认

任一电脑点击“检查健康”。预期类似：

```text
2 终端 / 0 会话 / 0 呼叫 / 运行 120 秒 / 事件 4/200
```

其中“运行”用于判断信令服务是否刚重启，“事件”用于判断内存事件日志当前条数和保留上限。

在 PC-B 也可用 PowerShell 检查端口：

```powershell
Test-NetConnection <PC-A局域网IP> -Port 7077
```

`TcpTestSucceeded` 应为 `True`。

## 七、测试用例

### 1. 示教室呼叫手术室

操作：

1. PC-B 在“信令目标”选择 `手术室端 PC-A`。
2. 发起模式选择“交互模式”。
3. 点击“信令呼叫选中终端”。
4. PC-A 查看待确认呼叫，接受确认选择“确认交互”。
5. PC-A 点击“接受呼叫”。

预期：

1. 两端均进入会话。
2. 最终模式为“交互模式”。
3. 参与数量为 `2 / <上限>`。
4. 默认显示通道 1。

### 2. 手术室呼叫示教室

操作：

1. PC-A 在“信令目标”选择 `示教室端 PC-B`。
2. 设置手术室参与上限，例如 `4`。
3. 点击“信令呼叫选中终端”。
4. PC-B 点击“接受呼叫”。

预期：

1. 两端进入会话。
2. `ownerEndpointId` 逻辑归属手术室端。
3. 参与上限按手术室端设置生效。

### 3. 发布订阅通道 WebRTC 音视频

操作：

1. PC-A 与 PC-B 建立信令会话。
2. PC-A 确认本端角色为“手术室端”。
3. PC-B 按需勾选通道 1 至通道 4，并选择单画面、双画面或四画面布局。
4. 双方按现场设备选择“音频输入设备”和“音频输出设备”；没有特殊声卡时保持系统默认输出。
5. PC-A 点击“发布订阅通道媒体”。如果订阅通道未预览，软件会自动启动对应通道预览；交互模式下会同时尝试采集本地麦克风。
6. PC-B 查看“状态”中的“媒体链路”“交互音频”和远端显示区。
7. 如 PC-B 黑屏、媒体诊断长期等待或 PC-A 刚点击过“停止媒体链路”，PC-B 点击“请求重发媒体”。

预期：

1. PC-A 状态显示正在发布或媒体链路已协商。
2. PC-B 状态显示正在接收或已收到手术室端媒体。
3. PC-B 远端显示区出现已订阅通道的动态画面。
4. 交互模式下，PC-B 应能收到 PC-A 远端音频；如果 PC-B 页面具备麦克风权限，PC-A 也应能收到 PC-B 远端音频。
5. PC-B 请求重发媒体后，PC-A 自动重新协商，PC-B 再次收到当前订阅通道画面。

边界说明：

当前验证手术室端向示教室端按订阅发布多通道模拟视频和基础音频。音频输出设备选择依赖浏览器 `setSinkId` 能力，不支持时自动回退系统默认输出。真实 USB 4 路采集卡并发、长稳运行、码率和资源占用仍需后续专项测试。

### 4. 仅收看模式确认

操作：

1. 任一端以“交互模式”发起呼叫。
2. 接收端接受前将“接受确认”改为“确认仅收看”。
3. 接收端点击“接受呼叫”。

预期：

1. 最终模式为“仅收看”。
2. “建立音频通话”按钮保持禁用。

### 5. 按需拉流和布局

操作：

1. 建立会话后，勾选通道 2、通道 3 或通道 4。
2. 切换“单画面”“双画面”“四画面”。

预期：

1. 远端显示区按订阅通道和布局变化。
2. 信令服务向会话参与方广播订阅变化。

边界说明：

发布订阅通道媒体后，已订阅通道应切换为远端 MediaStream；发布中继续增减订阅通道、观摩端加入、观摩端离会、普通观摩端异常断开、示教室端同 ID 重连或接收端请求重发媒体时，手术室端应自动重新协商媒体链路。未订阅通道仍可能显示本地模拟预览或空状态，不代表远端已经传输。

接收端点击“停止媒体链路”只停止本端到手术室 owner 的媒体链路；手术室端应只关闭该接收端对应的 PeerConnection。三端测试时，一个观摩端停止媒体后，其他示教室端或观摩端应继续显示已订阅远端画面。

### 6. 标注同步

操作：

1. PC-A 手术室端输入标注内容。
2. 勾选“手术室端标注远端可见”。

预期：

1. PC-B 远端显示区出现标注层。
2. PC-B 作为示教室端不能编辑标注控件。

### 7. 交互音频链路

操作：

1. 在交互模式会话中点击“发布订阅通道媒体”，或在媒体链路已建立后点击“建立音频通话”。
2. 浏览器请求麦克风权限时选择允许。
3. 观察双方“交互音频”和“音频诊断”状态。
4. 在“音频输出设备”中明确选择现场使用的扬声器或声卡。
5. 点击“停止音频”。

预期：

1. 状态显示本地音频轨道已建立、已加入 WebRTC，或已接收远端音频。
2. “音频诊断”显示安全上下文、输入/输出设备数量、是否支持选择输出设备，以及当前麦克风和回放设备。
3. “媒体诊断 / 连接诊断”中对应 PeerConnection 的详情显示 `音频 本地X 远端Y`，用于判断本端是否已发送音频轨道、是否已收到远端音频轨道。
4. 停止后本地音频状态回到“未建立”。

边界说明：

如果 PC-B 直接打开 `http://192.168.1.118:5173`，浏览器可能因为非安全 HTTP 来源禁止采集 PC-B 本地麦克风，此时 PC-B 可以接收手术室端音频，但不一定能向手术室端发言。需要双端发言时，PC-B 应本机运行前端并用 `http://127.0.0.1:5173` 打开，或改用 HTTPS / Electron 客户端。

出现黑屏、音频无声或延迟明显时，两端分别点击“复制诊断快照”，把快照文本保存到测试记录。已连接信令时，复制前会自动尝试刷新最近信令事件。快照用于比对本端 ID、信令连接、会话参与方、通道源模式、预览状态、音频诊断、媒体统计、结构化 WebRTC 指标、PeerConnection 摘要和最近信令事件；快照不包含信令令牌、设备 ID、SDP、ICE candidate 原文、患者信息或媒体数据。

如果音频能听到但延时明显，先排除以下因素：

1. 不使用蓝牙耳机、蓝牙音箱或电视扬声器，优先使用有线麦克风和有线扬声器。
2. 在“音频输出设备”中明确选择当前要使用的声卡、扬声器或会议平板回放设备，避免系统默认输出落到 HDMI、蓝牙或显示器音频。
3. 查看“音频诊断”：如果显示“非安全上下文”，该端可能只能接收远端音频，不能采集本地麦克风；如果输出数量为 `0` 或显示“系统默认输出”，优先检查浏览器和系统声卡设置。
4. 关闭 Windows 声音增强、空间音效、降噪软件和会议软件的虚拟音频处理。
5. 避免通过扩展坞、显示器 HDMI 音频或 USB Hub 转接音频设备。
6. 两台电脑保持有线局域网连接，避免 Wi-Fi 漫游和 VPN。
7. 手术室端发布 4K 采集画面时，如果音频仍被拖慢，应先降低视频采集分辨率或码率，再做正式低延迟测试。
8. 查看“状态 / 媒体统计”：如果视频发送码率长期为 `-`，优先排查发布端预览、WebRTC 协商和浏览器权限；如果接收码率长期为 `-`，优先排查接收端 ICE、网络和订阅通道；如果音频缓冲长期高于 200 ms，优先排查浏览器和音频设备；如果 RTT 明显偏高，优先排查网络；如果 ICE 路由显示 `-→-` 或不是预期的 `host`/`relay`，优先排查 STUN/TURN 配置和网络路径。

### 8. 会议参与上限

操作：

1. PC-A 与 PC-B 建立上限为 `2` 的会话。
2. 在任一电脑新开浏览器标签页，使用新 ID 注册为 `观摩端`，例如 `observer-1`。
3. 从“会话目录”选择当前会话，点击“加入信令会话”。

预期：

1. 服务端返回参与上限错误。
2. 前端显示“已达到参与上限”。
3. 观摩端不会进入该会话。

### 9. 会话目录加入

操作：

1. PC-A 与 PC-B 建立上限为 `3` 或以上的会话。
2. 新开第三个浏览器标签页，注册为 `观摩端`。
3. 点击“刷新会话”。
4. 从“会话目录”选择已有会话并加入。

预期：

1. 观摩端进入会话。
2. 原两端参与数量更新为 `3 / <上限>`。
3. 参与方列表包含观摩端。

### 10. 断线清理

操作：

1. 建立交互会话。
2. 点击“建立音频通话”。
3. 在 PC-A 停止信令服务器，或关闭两人会话中的其中一端浏览器。

预期：

1. 前端清理会话状态。
2. 在线目录和信令目标被清理。
3. 本地音频状态回到“未建立”。
4. 健康检查状态回到 `-`。

多人会话补充验证：

1. PC-A 与 PC-B 建立上限为 `3` 或以上的会话，并让观摩端加入。
2. PC-A 发布订阅通道媒体。
3. 直接关闭普通观摩端浏览器。

预期：

1. PC-A 与 PC-B 会话仍保留，参与人数回到 `2 / <上限>`。
2. 普通观摩端从参与方和订阅中移除。
3. PC-A 自动重新协商媒体链路，PC-B 远端画面继续在线。

同 ID 重连补充验证：

1. PC-A 与 PC-B 建立会话并发布订阅通道媒体。
2. 在 PC-B 新开浏览器标签页或重新打开客户端，使用与原 PC-B 相同的“本端 ID”连接信令。

预期：

1. 原 PC-B 页面提示 endpointId 被接管或信令断开。
2. 新 PC-B 页面恢复到原会话。
3. PC-A 自动重新协商媒体链路，新 PC-B 页面不需要手动点击发布即可重新看到远端画面。

### 11. 多显示器扩展窗口

操作：

1. 在有多显示器的电脑上建立会话。
2. 点击“刷新显示器”。
3. 在“扩展显示器”选择目标显示器。
4. 点击远端画面中的“扩展窗口”。

预期：

1. 打开独立扩展窗口。
2. 状态提示目标显示器。
3. 窗口可用于拖动或摆放到示教显示屏。

### 12. USB 摄像机云台和镜头控制

操作：

1. 在 PC-A 将目标通道切换为“USB/摄像头设备”。
2. 选择目标 USB 摄像机并点击“预览”。
3. 点击该通道下方的方向和变倍按钮。

预期：

1. 支持浏览器 UVC PTZ 能力的摄像机可响应方向或变倍控制。
2. 不支持 UVC PTZ 的采集卡、摄像机或浏览器应显示明确的不支持提示。

边界说明：

当前云台控制仅验证浏览器可调用的 UVC `pan`、`tilt`、`zoom` 能力；厂商私有 USB 控制协议、串口控制、VISCA、Pelco 或专用 SDK 仍需后续专项接入。

### 13. 患者绑定和录像索引

操作：

1. 在 PC-A 输入 `HIS-001`。
2. 点击“模拟 HIS 查询”。
3. 启动预览并开始选中通道录制。
4. 停止录制。

预期：

1. 录像索引生成新记录。
2. 记录显示患者名称和 HIS ID。
3. 可按文件、通道、患者姓名或 HIS ID 搜索录像，并执行定位、导出、删除和加入 AI 队列。
4. Electron 客户端配置 FTP 环境变量后，可点击“上传FTP”并看到上传成功的远端路径；未配置时应显示明确失败原因。

边界说明：

当前 HIS 为本地模拟数据，不连接医院真实 HIS。FTP 上传为 PoC 级文件发送能力，不包含断点续传、队列重试、归档生命周期、上传审计和医院文件服务权限模型。

http://<PC-A局域网IP>:7077/health
http://<PC-A局域网IP>:7077/directory
http://<PC-A局域网IP>:7077/sessions
http://<PC-A局域网IP>:7077/events

如果启用了 `SIGNALING_AUTH_TOKEN`，`/directory`、`/sessions` 和 `/events` 需要携带令牌。

## 九、常见问题处理

### 1. PC-B 无法连接信令服务器

检查项：

1. PC-A 信令服务器是否使用 `SIGNALING_HOST=0.0.0.0` 启动。
2. PC-A 防火墙是否开放 TCP `7077`。
3. 两台电脑是否在同一局域网。
4. PC-B 执行 `Test-NetConnection <PC-A局域网IP> -Port 7077` 是否成功。
5. 信令地址是否写成 `ws://<PC-A局域网IP>:7077/signal`，不要使用 PC-B 自己的 `127.0.0.1`。

### 2. 在线目录看不到对方

检查项：

1. 两端是否都点击了“连接信令”。
2. 两端 `本端 ID` 是否重复。
3. 如果启用令牌，两端令牌是否一致。
4. 是否连接到了同一个 PC-A 信令地址。

### 3. 提示信令鉴权失败

原因通常是 `SIGNALING_AUTH_TOKEN` 已启用，但前端“信令令牌”为空或不一致。统一令牌后重新连接。

### 4. 摄像头或麦克风不可用

检查项：

1. 是否使用 `http://127.0.0.1:5173` 打开前端。
2. 浏览器是否授权摄像头和麦克风。
3. Windows 隐私设置是否允许桌面应用访问摄像头和麦克风。
4. 如果从另一台电脑访问 `http://<IP>:5173`，浏览器可能因非安全上下文阻止媒体权限。

### 5. 示教室端看不到真实手术室视频

当前阶段已支持按订阅 WebRTC P2P 视频。如果 PC-A 点击“发布订阅通道媒体”后 PC-B 仍黑屏，按以下顺序检查：

1. 两端是否已经进入同一个信令会话。
2. PC-A 本端角色是否为“手术室端”；示教室端不能发布媒体。
3. PC-A 被订阅通道是否能本地预览。
4. PC-B “状态 / 媒体链路”是否显示正在接收或已收到手术室端媒体。
5. PC-A 点击“刷新事件”，确认存在从 PC-A 到 PC-B 的 `peer.signal.forwarded` / `media-offer`，并且摘要里有订阅通道和轨道数；再确认有 PC-B 到 PC-A 的 `media-answer`。如果只有呼叫事件，没有 `media-offer`，优先检查 PC-A 是否点击了“发布订阅通道媒体”以及本端角色是否为手术室端。
6. 两台电脑是否在同一局域网；当前 PoC 默认未配置 STUN/TURN，不保证跨 NAT、VPN 或复杂路由可用。如需跨网段联调，应先在 `public/config.json` 配置 `webrtc.iceServers`。
7. 浏览器控制台是否出现 `WebRTC 协商失败`、`ICE` 或 `RTCPeerConnection` 相关错误。
8. 两端分别点击“复制诊断快照”，确认 PC-A 快照的 `media.peerConnections` 有 PC-B，PC-B 快照的 `media.stats` 和 `media.statsMetrics` 有视频码率、包计数或 ICE 路由，`recentEvents` 中存在 `media-offer`、`media-answer` 和 `ice` 摘要。

如果发布中继续勾选通道 2 至通道 4 后仍看不到真实远端画面，优先检查 PC-B“媒体诊断”是否仍停留在“等待远端”，以及“连接诊断”里的连接、ICE 和协商状态；正常情况下 PC-A 不需要再次点击“发布订阅通道媒体”。

### 6. 点击加入会话提示达到参与上限

检查手术室端设置的参与上限。若上限为 `2`，第三个终端会被拒绝；需要测试观摩端加入时，应将上限设置为 `3` 或更高。

## 十、测试完成标准

本手册测试完成应满足：

1. 两端均能连接同一信令服务器并在在线目录看到对方。
2. 示教室端呼叫手术室端、手术室端呼叫示教室端均可建立会话。
3. 交互模式和仅收看模式均符合预期。
4. 手术室端发布订阅通道媒体后，示教室端能看到已订阅通道的真实远端动态画面。
5. 通道订阅、布局切换、标注和参与上限提示可用。
6. 断线后会话、在线目录、目标选择、健康状态、本地音频轨道和 WebRTC 媒体链路被清理。
7. 可完成至少一次模拟 HIS 绑定录像，并在录像索引中查看患者元数据。
8. 测试人员明确记录当前仍不包含真实 USB 4 路长稳验收、SFU、RTSP/SRT、手机直播和生产级音频回声消除验收。
