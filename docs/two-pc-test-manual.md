# 两台电脑手术室与示教室模拟测试使用手册

## 一、适用范围

本文档用于在两台 Windows 电脑上验证当前阶段手术示教软件 PoC 的呼叫、互动、会话目录、按需拉流控制、标注、会议上限、断线清理和基础录像流程。

当前阶段验证的是 C/S 控制面和本地媒体模拟流程，不包含真实跨终端音视频传输。手术室端 USB 采集卡画面不会通过网络传输到示教室端；示教室端看到的远端画面仍由本地模拟预览流承载。真实 WebRTC/SFU、RTSP/SRT、Android 会议平板和手机直播属于下一阶段。

## 二、本阶段开发成果汇总

1. 完成 4 路本地预览、选择性录制、录像索引、基础回放、导出和删除。
2. 完成 WebSocket 信令服务，支持终端注册、在线目录、呼叫、接受、拒绝、取消、会话建立、会话结束、离会和会话目录发现。
3. 支持手术室端与示教室端双向发起呼叫，支持交互模式和仅收看模式确认。
4. 支持会议参与上限，超限时服务端拒绝加入并在前端提示。
5. 支持手术室端会话 owner 归属，多人会议中非 owner 不能结束整场会议。
6. 支持按需订阅通道、布局切换、标注广播和 `peer.signal` 协商消息透传。
7. 支持可选共享令牌门禁、HTTP 健康检查、会话摘要目录和内存事件日志。
8. 支持信令忙线保护、待处理呼叫超时、重复 endpointId 接管、会话恢复、协议错误分支和 WebSocket 心跳清理。
9. 支持信令断开时清理会话、在线目录、目标选择、健康状态和本地交互音频轨道。
10. 支持远端画面独立扩展窗口；Electron 模式下可枚举 Windows 显示器并定位到指定显示器。
11. 已接入 GitHub Actions CI，`npm run verify` 覆盖构建、高危依赖审计、前端烟测和信令契约测试。

## 三、测试拓扑

推荐拓扑如下：

| 角色 | 电脑 | 运行内容 | 访问地址 |
|---|---|---|---|
| 手术室端 | PC-A | 信令服务器、前端客户端 | `http://127.0.0.1:5173` |
| 示教室端 | PC-B | 前端客户端 | `http://127.0.0.1:5173` |
| 信令服务 | PC-A | WebSocket 服务 | `ws://<PC-A局域网IP>:7077/signal` |

推荐两台电脑都本地运行前端。原因是浏览器对摄像头、麦克风等媒体权限通常要求安全上下文，`127.0.0.1` 比直接访问另一台电脑的 `http://<IP>:5173` 更稳定。

## 四、测试前准备

### 1. 软件环境

两台电脑均需要：

1. Windows 10 或 Windows 11。
2. Node.js 与 npm。
3. Chrome 或 Edge。
4. 本仓库代码和依赖。

在两台电脑的仓库目录执行：

```powershell
cd D:\CodeX\UST\phase1-poc
npm install
```

如果已经安装过依赖，可以跳过 `npm install`。

### 2. 获取手术室端电脑 IP

在 PC-A 执行：

```powershell
ipconfig
```

记录当前局域网网卡的 IPv4 地址，例如：

```text
192.168.10.21
```

后文用 `<PC-A局域网IP>` 表示该地址。

### 3. 开放信令端口

在 PC-A 以管理员身份打开 PowerShell，执行：

```powershell
New-NetFirewallRule -DisplayName "UST Signaling 7077" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 7077
```

如果需要从 PC-B 直接访问 PC-A 的前端页面，还需要开放 `5173`：

```powershell
New-NetFirewallRule -DisplayName "UST Web 5173" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 5173
```

推荐测试方式不要求开放 `5173`，因为两台电脑各自运行本地前端。

## 五、启动步骤

### 1. 在 PC-A 启动信令服务器

在 PC-A 执行：

```powershell
cd D:\CodeX\UST\phase1-poc
$env:SIGNALING_HOST = "0.0.0.0"
$env:SIGNALING_PORT = "7077"
npm run server:signaling
```

正常输出应包含：

```text
signaling server listening on ws://0.0.0.0:7077/signal
```

如果需要启用共享令牌门禁：

```powershell
$env:SIGNALING_AUTH_TOKEN = "test-token"
```

启用令牌后，两台前端都必须填写同一个信令令牌。

### 2. 在 PC-A 启动手术室端前端

在 PC-A 新开一个 PowerShell 窗口：

```powershell
cd D:\CodeX\UST\phase1-poc
npm run dev:web
```

浏览器打开：

```text
http://127.0.0.1:5173
```

### 3. 在 PC-B 启动示教室端前端

在 PC-B 执行：

```powershell
cd D:\CodeX\UST\phase1-poc
npm run dev:web
```

浏览器打开：

```text
http://127.0.0.1:5173
```

### 4. 可选：只在 PC-A 托管前端页面

如果 PC-B 不安装仓库，可以在 PC-A 用局域网方式启动前端：

```powershell
cd D:\CodeX\UST\phase1-poc
npx vite --host 0.0.0.0 --port 5173
```

PC-B 浏览器打开：

```text
http://<PC-A局域网IP>:5173
```

该方式适合验证信令控制流，不推荐用于摄像头和麦克风测试，因为浏览器可能阻止非安全 HTTP 来源的媒体权限。

## 六、客户端配置

### 1. PC-A 手术室端配置

在前端“信令控制面”中填写：

| 字段 | 建议值 |
|---|---|
| 信令地址 | `ws://<PC-A局域网IP>:7077/signal` |
| 信令令牌 | 为空；如启用门禁则填写 `test-token` |
| 本端 ID | `or-room-1` |
| 本端名称 | `手术室端 PC-A` |
| 本端角色 | `手术室端` |

点击“连接信令”。预期显示“已注册 手术室端 PC-A”。

### 2. PC-B 示教室端配置

在前端“信令控制面”中填写：

| 字段 | 建议值 |
|---|---|
| 信令地址 | `ws://<PC-A局域网IP>:7077/signal` |
| 信令令牌 | 为空；如启用门禁则填写 `test-token` |
| 本端 ID | `teach-room-1` |
| 本端名称 | `示教室端 PC-B` |
| 本端角色 | `示教室端` |

点击“连接信令”。预期显示“已注册 示教室端 PC-B”。

### 3. 连通性确认

任一电脑点击“检查健康”。预期类似：

```text
2 终端 / 0 会话 / 0 呼叫
```

在 PC-B 也可用 PowerShell 检查端口：

```powershell
Test-NetConnection <PC-A局域网IP> -Port 7077
```

`TcpTestSucceeded` 应为 `True`。

## 七、测试用例

### 1. 示教室呼叫手术室

操作：

1. PC-B 在“信令目标”选择 `手术室端 PC-A`。
2. 发起模式选择“交互模式”。
3. 点击“信令呼叫选中终端”。
4. PC-A 查看待确认呼叫，接受确认选择“确认交互”。
5. PC-A 点击“接受呼叫”。

预期：

1. 两端均进入会话。
2. 最终模式为“交互模式”。
3. 参与数量为 `2 / <上限>`。
4. 默认显示通道 1。

### 2. 手术室呼叫示教室

操作：

1. PC-A 在“信令目标”选择 `示教室端 PC-B`。
2. 设置手术室参与上限，例如 `4`。
3. 点击“信令呼叫选中终端”。
4. PC-B 点击“接受呼叫”。

预期：

1. 两端进入会话。
2. `ownerEndpointId` 逻辑归属手术室端。
3. 参与上限按手术室端设置生效。

### 3. 仅收看模式确认

操作：

1. 任一端以“交互模式”发起呼叫。
2. 接收端接受前将“接受确认”改为“确认仅收看”。
3. 接收端点击“接受呼叫”。

预期：

1. 最终模式为“仅收看”。
2. “建立音频通话”按钮保持禁用。

### 4. 按需拉流和布局

操作：

1. 建立会话后，勾选通道 2、通道 3 或通道 4。
2. 切换“单画面”“双画面”“四画面”。

预期：

1. 远端显示区按订阅通道和布局变化。
2. 信令服务向会话参与方广播订阅变化。

边界说明：

当前画面是本地模拟预览流，不是真实从另一台电脑传输过来的画面。

### 5. 标注同步

操作：

1. PC-A 手术室端输入标注内容。
2. 勾选“手术室端标注远端可见”。

预期：

1. PC-B 远端显示区出现标注层。
2. PC-B 作为示教室端不能编辑标注控件。

### 6. 交互音频约束

操作：

1. 在交互模式会话中点击“建立音频通话”。
2. 浏览器请求麦克风权限时选择允许。
3. 点击“停止音频”。

预期：

1. 状态显示本地音频轨道已建立。
2. 停止后状态回到“未建立”。

边界说明：

当前只验证本地浏览器音频采集约束，包括回声消除、噪声抑制和自动增益控制；不传输远端音频。

### 7. 会议参与上限

操作：

1. PC-A 与 PC-B 建立上限为 `2` 的会话。
2. 在任一电脑新开浏览器标签页，使用新 ID 注册为 `观摩端`，例如 `observer-1`。
3. 从“会话目录”选择当前会话，点击“加入信令会话”。

预期：

1. 服务端返回参与上限错误。
2. 前端显示“已达到参与上限”。
3. 观摩端不会进入该会话。

### 8. 会话目录加入

操作：

1. PC-A 与 PC-B 建立上限为 `3` 或以上的会话。
2. 新开第三个浏览器标签页，注册为 `观摩端`。
3. 点击“刷新会话”。
4. 从“会话目录”选择已有会话并加入。

预期：

1. 观摩端进入会话。
2. 原两端参与数量更新为 `3 / <上限>`。
3. 参与方列表包含观摩端。

### 9. 断线清理

操作：

1. 建立交互会话。
2. 点击“建立音频通话”。
3. 在 PC-A 停止信令服务器，或关闭其中一端浏览器。

预期：

1. 前端清理会话状态。
2. 在线目录和信令目标被清理。
3. 本地音频状态回到“未建立”。
4. 健康检查状态回到 `-`。

### 10. 多显示器扩展窗口

操作：

1. 在有多显示器的电脑上建立会话。
2. 点击“刷新显示器”。
3. 在“扩展显示器”选择目标显示器。
4. 点击远端画面中的“扩展窗口”。

预期：

1. 打开独立扩展窗口。
2. 状态提示目标显示器。
3. 窗口可用于拖动或摆放到示教显示屏。

### 11. 患者绑定和录像索引

操作：

1. 在 PC-A 输入 `HIS-001`。
2. 点击“模拟 HIS 查询”。
3. 启动预览并开始选中通道录制。
4. 停止录制。

预期：

1. 录像索引生成新记录。
2. 记录显示患者名称和 HIS ID。
3. 可执行定位、导出、删除和加入 AI 队列。

边界说明：

当前 HIS 为本地模拟数据，不连接医院真实 HIS。

## 九、常见问题处理

### 1. PC-B 无法连接信令服务器

检查项：

1. PC-A 信令服务器是否使用 `SIGNALING_HOST=0.0.0.0` 启动。
2. PC-A 防火墙是否开放 TCP `7077`。
3. 两台电脑是否在同一局域网。
4. PC-B 执行 `Test-NetConnection <PC-A局域网IP> -Port 7077` 是否成功。
5. 信令地址是否写成 `ws://<PC-A局域网IP>:7077/signal`，不要使用 PC-B 自己的 `127.0.0.1`。

### 2. 在线目录看不到对方

检查项：

1. 两端是否都点击了“连接信令”。
2. 两端 `本端 ID` 是否重复。
3. 如果启用令牌，两端令牌是否一致。
4. 是否连接到了同一个 PC-A 信令地址。

### 3. 提示信令鉴权失败

原因通常是 `SIGNALING_AUTH_TOKEN` 已启用，但前端“信令令牌”为空或不一致。统一令牌后重新连接。

### 4. 摄像头或麦克风不可用

检查项：

1. 是否使用 `http://127.0.0.1:5173` 打开前端。
2. 浏览器是否授权摄像头和麦克风。
3. Windows 隐私设置是否允许桌面应用访问摄像头和麦克风。
4. 如果从另一台电脑访问 `http://<IP>:5173`，浏览器可能因非安全上下文阻止媒体权限。

### 5. 示教室端看不到真实手术室视频

这是当前阶段的预期边界。当前版本只验证控制流和本地媒体模拟，不传输真实跨机音视频。真实远端画面需要下一阶段接入 WebRTC/SFU、WHEP 或其他媒体服务。

### 6. 点击加入会话提示达到参与上限

检查手术室端设置的参与上限。若上限为 `2`，第三个终端会被拒绝；需要测试观摩端加入时，应将上限设置为 `3` 或更高。

## 十、测试完成标准

本手册测试完成应满足：

1. 两端均能连接同一信令服务器并在在线目录看到对方。
2. 示教室端呼叫手术室端、手术室端呼叫示教室端均可建立会话。
3. 交互模式和仅收看模式均符合预期。
4. 通道订阅、布局切换、标注和参与上限提示可用。
5. 断线后会话、在线目录、目标选择、健康状态和本地音频轨道被清理。
6. 可完成至少一次模拟 HIS 绑定录像，并在录像索引中查看患者元数据。
7. 测试人员明确记录当前不包含真实跨终端音视频传输。
