# Phase 2 阶段状态与边界

## 一、当前结论

Phase 2 已形成可运行、可自动化验证的呼叫与互动控制面 PoC。当前版本适合继续进入真实媒体服务、跨终端联调和后续业务集成设计，但不能直接视为生产级手术示教系统。

## 二、已实现范围

1. 4 路本地预览、选择性录制、录像索引和基础回放。
2. WebSocket 信令服务器，支持终端注册、通讯录、呼叫、接受、拒绝、会话创建、会话结束和断连清理。
3. 终端注册时上报视频通道清单，通讯录返回终端绑定的通道元数据。
4. 前端可连接信令服务器，注册本端，选择在线目标，发起信令呼叫。
5. 前端可接收远端信令呼入，并执行接受或拒绝。
6. 会话支持交互模式和仅收看模式，任一方选择仅收看时最终会话为仅收看。
7. 会议参与上限按手术室端设置解析，反向呼叫时仍保留手术室端上限。
8. 会话默认订阅通道 1，并支持通道订阅更新。
9. 观摩端加入会话后，前端参与人数和参与方列表可随信令更新。
10. 标注文本和可见状态可通过信令会话广播。
11. 信令服务支持会话内 `peer.signal` 透传，用于后续 WebRTC offer、answer 和 ICE 候选协商。
12. 交互模式下可建立本地音频采集轨道，并启用浏览器侧回声消除、噪声抑制和自动增益约束。
13. 前端启动时读取 `public/config.json` 作为默认信令地址和本端终端信息。
14. 前端可调用信令 `/health`，显示终端数量、会话数量和待处理呼叫数量。
15. 非法信令地址会显示可见错误状态，不抛出未处理异常。
16. 远端画面可打开独立扩展窗口，作为多显示器扩展的前置 PoC。
17. GitHub Actions CI 已接入构建、依赖审计、前端烟测和信令契约测试。

## 三、自动化验证

本地统一验证命令：

```powershell
npm run verify
```

该命令包含：

1. `npm run build`
2. `npm audit --audit-level=high`
3. `npm run test:smoke`
4. `npm run test:signaling`

当前自动化覆盖：

1. 页面渲染、4 路模拟预览。
2. 短录制、停止录制、录像索引生成。
3. 本地模拟呼叫、接受、布局切换、标注和人数上限提示。
4. 交互音频轨道建立和停止。
5. 前端主动信令呼叫并进入已接受会话。
6. 远端信令呼入，前端接受。
7. 远端信令呼入，前端拒绝。
8. 观摩端加入会话后前端状态更新。
9. 标注通过信令广播给远端。
10. 会话显式结束。
11. 终端异常断开后的会话清理。
12. 终端通道清单注册和目录返回。
13. 会话内 WebRTC 协商消息透传。
14. 前端信令通道订阅更新。
15. HTTP 健康检查和前端健康检查。
16. 非法信令地址错误提示。
17. 远端画面扩展窗口。

## 四、未实现边界

以下能力尚未实现，不能在交付、演示或招标材料中描述为已完成：

1. 真实跨终端音视频媒体传输、WebRTC/SFU、WHEP、RTSP/SRT 接入和媒体转码。
2. Android 会议平板端原生客户端。
3. 手机端直播分发和多并发直播服务。
4. USB 摄像机云台、镜头控制和厂商 SDK 适配。
5. 真实 HIS 患者信息获取、接口鉴权和医院字段映射。
6. FTP 上传、移动存储导出流程和文件归档策略。
7. 终端鉴权、TLS、审计、权限模型和操作日志。
8. 标注权限控制、版本控制和录像回放绑定。
9. Windows 多显示器独立扩展窗口。
10. 长时间真实采集卡压力测试和医院网络抖动测试。

## 五、建议下一阶段

下一阶段不应继续堆叠前端模拟效果，应优先验证真实媒体链路：

1. 建立媒体服务 PoC，明确 WebRTC/SFU 与 RTSP/SRT 接入边界。
2. 选定一种低延迟远端预览协议，并验证手术室端 4 路发布、示教室端按需订阅。
3. 在真实 USB 采集卡和医院局域网环境中复测延迟、丢包恢复、音频回声消除和长时间稳定性。
4. 将信令会话 ID 与媒体房间 ID、通道 ID 建立明确映射。
5. 再进入 HIS、文件归档、FTP、AI 接口和 Android 端开发。
