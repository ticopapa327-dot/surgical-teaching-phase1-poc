# Phase 2 阶段状态与边界

## 一、当前结论

Phase 2 已形成可运行、可自动化验证的呼叫与互动控制面 PoC。当前版本适合继续进入真实媒体服务、跨终端联调和后续业务集成设计，但不能直接视为生产级手术示教系统。

## 二、已实现范围

1. 4 路本地预览、选择性录制、录像索引和基础回放。
2. WebSocket 信令服务器，支持终端注册、通讯录、呼叫、接受、拒绝、取消、会话创建、会话结束和断连清理。
3. 终端注册时上报视频通道清单，通讯录返回终端绑定的通道元数据。
4. 前端可按本端角色上报能力和通道，观摩端不会默认声明手术室采集通道。
5. 前端可连接信令服务器，注册本端，选择在线目标，发起信令呼叫。
6. 前端可接收远端信令呼入，并执行接受或拒绝。
7. 会话支持交互模式和仅收看模式，任一方选择仅收看时最终会话为仅收看。
8. 会议参与上限和会话控制主体按手术室端解析，反向呼叫时仍保留手术室端上限和 owner。
9. 会话默认订阅通道 1，并支持通道订阅更新。
10. 观摩端加入或离开会话后，前端参与人数和参与方列表可随信令更新。
11. 前端支持输入会话 ID 加入既有信令会话，也支持本端离开会话而不强制结束剩余会议；多人会话中非 owner 不能结束整场会议，owner 离会会结束该会议。
12. 前端可读取信令会话目录，从目录选择可加入会议，也可手工输入会话 ID。
13. 标注文本和可见状态可通过信令会话广播，前端仅允许手术室角色操作标注开关，服务端拒绝非手术室端标注更新。
14. 信令服务支持会话内 `peer.signal` 透传，用于后续 WebRTC offer、answer 和 ICE 候选协商。
15. 信令服务对终端注册、订阅通道和参与人数上限进行基础归一化，重复 endpointId 注册会由新连接替换旧连接，并可向新连接恢复既有会话状态。
16. 信令服务具备忙线保护、待处理呼叫超时和 WebSocket 心跳清理，任一端已有待处理呼叫或处于会话中时不会创建新的待处理呼叫，也不会被加入另一个会话；未确认呼叫超时后自动取消，心跳无响应连接会按断线清理。
17. 信令服务支持可选共享令牌门禁，错误令牌不会完成终端注册；启用后在线目录和会话目录 HTTP 接口也需要令牌；该能力不等同于生产级鉴权。
18. 信令服务提供内存型控制面事件日志，用于调试和验收追踪；该能力不等同于生产级审计。
19. 交互模式下可建立本地音频采集轨道，并启用浏览器侧回声消除、噪声抑制和自动增益约束。
20. 前端启动时读取 `public/config.json` 作为默认信令地址和本端终端信息；连接成功后锁定本端注册身份字段，避免会话中身份漂移。
21. 前端可调用信令 `/health`，显示终端数量、会话数量和待处理呼叫数量。
22. 非法信令地址会显示可见错误状态，不抛出未处理异常。
23. 远端画面可打开独立扩展窗口；Electron 模式下可枚举 Windows 显示器并按目标显示器定位弹窗，浏览器模式下回退为默认弹窗。
24. GitHub Actions CI 已接入构建、依赖审计、前端烟测和信令契约测试。

## 三、自动化验证

本地统一验证命令：

```powershell
npm run verify
```

该命令包含：

1. `npm run build`
2. `npm audit --audit-level=high`
3. `npm run test:smoke`
4. `npm run test:signaling`

当前自动化覆盖：

1. 页面渲染、4 路模拟预览。
2. 短录制、停止录制、录像索引生成。
3. 本地模拟呼叫、接受、布局切换、标注和人数上限提示。
4. 交互音频轨道建立和停止。
5. 前端主动信令呼叫并进入已接受会话。
6. 远端信令呼入，前端接受。
7. 远端信令呼入，前端拒绝。
8. 观摩端加入会话后前端状态更新。
9. 前端输入会话 ID 加入既有信令会话。
10. 前端从信令会话目录选择会话并加入。
11. 前端离开信令会话后，剩余参会方收到参与列表更新。
12. 标注通过信令广播给远端。
13. 会话显式结束。
14. 终端异常断开和前端主动断开后的会话及交互音频清理。
15. 终端通道清单注册和目录返回。
16. 会话内 WebRTC 协商消息透传。
17. 前端信令通道订阅更新。
18. 终端注册、订阅通道和参与人数上限归一化。
19. 重复 endpointId 注册替换旧连接。
20. endpointId 被接管后的前端提示和会话恢复。
21. 服务端忙线保护和前端忙线提示。
22. 待处理呼叫超时取消、心跳断连清理和 pendingCalls 归零。
23. HTTP 健康检查、会话摘要目录和前端健康检查。
24. 可选共享令牌门禁和前端错误令牌提示。
25. 控制面事件日志、令牌保护和令牌不落日志。
26. 前端按角色上报能力和通道，观摩端不声明发布通道。
27. 非法信令地址错误提示。
28. 远端画面扩展窗口和 Electron 显示器目标选择。
29. 自呼叫拒绝、未注册目录读取限制、协议错误分支、服务端模式合成和多人会话结束权限。

## 四、未实现边界

以下能力尚未实现，不能在交付、演示或招标材料中描述为已完成：

1. 真实跨终端音视频媒体传输、WebRTC/SFU、WHEP、RTSP/SRT 接入和媒体转码。
2. Android 会议平板端原生客户端。
3. 手机端直播分发和多并发直播服务。
4. 非 UVC 云台控制、厂商私有协议和厂商 SDK 适配。
5. 真实 HIS 患者信息获取、接口鉴权和医院字段映射。
6. FTP 上传、移动存储导出流程和文件归档策略。
7. 生产级终端鉴权、TLS、审计、权限模型和操作日志；当前共享令牌和事件日志只作为 PoC 门禁与调试追踪。
8. 生产级细粒度标注权限、标注版本控制和录像回放绑定；当前仅实现手术室角色限制。
9. 生产级多屏布局配置、启动恢复和窗口生命周期管理。
10. 长时间真实采集卡压力测试和医院网络抖动测试。

## 五、建议下一阶段

下一阶段不应继续堆叠前端模拟效果，应优先验证真实媒体链路：

1. 建立媒体服务 PoC，明确 WebRTC/SFU 与 RTSP/SRT 接入边界。
2. 选定一种低延迟远端预览协议，并验证手术室端 4 路发布、示教室端按需订阅。
3. 在真实 USB 采集卡和医院局域网环境中复测延迟、丢包恢复、音频回声消除和长时间稳定性。
4. 将信令会话 ID 与媒体房间 ID、通道 ID 建立明确映射。
5. 再进入 HIS、文件归档、FTP、AI 接口和 Android 端开发。
